TRACE Protocol

Open standard for auditable autonomous systems

Version 1.0.0 • RFC-2025-001 • Apache-2.0 License

Protocol Specification

The TRACE Protocol defines a standardized approach to recording, validating, and evidencing autonomous system actions through a unified Action → Policy → Evidence model.

Core Requirements

  • Vendor-neutral, model-agnostic HTTP API specification
  • Portable policy definitions with enforce and observe modes
  • Cryptographically verifiable evidence records
  • Human-in-the-loop approval workflows

Protocol Flow

// Action → Policy → Evidence
const decision = await trace.propose({
  type: "send_email",
  actor: { kind: "agent", name: "mail-bot" },
  target: "mailto:client@acme.com",
  params: { subject: "Proposal" }
});

if (decision.status === "requires_approval") {
  await fetch(`/approve/${decision.actionId}`, { 
    method: "POST" 
  });
}

await trace.evidence(decision.actionId, [
  { name: "email_sent", pass: true, note: "msgId=123" }
]);

Data Models

The protocol defines three core data structures that form the foundation of the Action → Policy → Evidence model.

Action

Structured description of autonomous system intent

Required Fields:

  • • id - Unique identifier
  • • type - Action type
  • • actor - Entity performing action
  • • target - Action target
  • • params - Action parameters
{
  "id": "a_123",
  "type": "send_email",
  "actor": { "kind": "agent", "name": "mail-bot" },
  "target": "mailto:client@acme.com",
  "params": { "subject": "Hi" },
  "timestamp": "2025-10-10T12:34:56Z"
}

Policy

Rules governing action execution

Policy Modes:

  • • enforce - Block until requirements met
  • • observe - Log but allow execution
  • • require - List of required checks
rules:
  - when: { actionType: "send_email" }
    require: ["reviewer_approval"]
    mode: enforce

Evidence

Verifiable proof of action outcomes

Evidence Types:

  • • checks - Policy compliance results
  • • artifacts - Generated outputs
  • • signatures - Cryptographic proofs
{
  "actionId": "a_123",
  "checks": [{ 
    "name": "reviewer_approval", 
    "pass": true, 
    "approver": "@fernando" 
  }],
  "timestamp": "2025-10-10T12:40:00Z"
}

API Reference

HTTP endpoints for implementing TRACE Protocol compliance

POST /propose

Submit an action for policy evaluation

curl -X POST /propose \
  -H "Content-Type: application/json" \
  -d '{
    "type": "send_email",
    "actor": { "kind": "agent", "name": "mail-bot" },
    "target": "mailto:client@acme.com",
    "params": { "subject": "Proposal" }
  }'

POST /evidence

Record evidence for a completed action

curl -X POST /evidence \
  -H "Content-Type: application/json" \
  -d '{
    "actionId": "a_123",
    "checks": [{ "name": "email_sent", "pass": true }]
  }'

GET /actions/{id}

Retrieve action details and status

curl /actions/a_123

Response:
{
  "id": "a_123",
  "status": "approved",
  "policy": { "mode": "enforce" },
  "evidence": [...]
}

POST /approve/{id}

Approve a pending action

curl -X POST /approve/a_123 \
  -H "Content-Type: application/json" \
  -d '{
    "approver": "user@example.com",
    "reason": "Manual approval"
  }'

TRACE Manifesto

Autonomy demands accountability. Agents must be able to act — and we must be able to verify why and how they acted. TRACE provides a simple, open standard to create trusted records for autonomous computation and events.

1) Purpose

Enable any organization to adopt AI agents without sacrificing safety, auditability, or velocity. TRACE is the connective tissue between intent, policy, and proof.

2) Scope

  • Actions: structured descriptions of what an agent intends to do.
  • Policies: constraints that determine if, when, and how actions may proceed.
  • Evidence: cryptographically signed or otherwise attestable records of what happened.

3) Principles

  • Open by default — Apache-2.0 license, public RFCs, community governance.
  • Model & vendor agnostic — works with any LLM/agent runtime.
  • Human-in-the-loop — approvals and overrides are first-class policies.
  • Minimal primitives — small surface area, pluggable storage and identity.
  • Provability — evidence is independently verifiable whenever possible.

4) How it works

Action → Policy → Evidence

1. Propose an Action (type, actor, target, params).
2. Apply Policy (enforce or observe; may require approval).
3. Emit Evidence (checks, results, artifacts, signatures).
4. Persist to an evidence store (database, ledger, or both).

5) Governance

TRACE Labs stewards the reference server, SDKs, and the RFC process. Substantial changes land through open proposals, community review, and versioned specs.

6) Non-goals

  • Not a full agent framework or LLM runtime.
  • Not a centralized trust authority.
  • Not tied to any specific storage, identity, or cloud.

7) License

Apache-2.0 for code and reference implementations; the spec is CC BY 4.0.

8) Call to action

Adopt the primitives, propose improvements, and contribute integrations. Build agents that leave a trail you can trust.

Governance

Open governance model for protocol evolution and standardization

RFC Process

Protocol changes follow a formal Request for Comments (RFC) process:

  1. Draft RFC with problem statement and proposed solution
  2. Community review period (minimum 30 days)
  3. Implementation feedback and testing
  4. Consensus decision by maintainers
  5. Versioned specification update
